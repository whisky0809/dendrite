# Dendrite v2 — Context Engine Plugin

> Topic-aware context assembly — viewport, not container.

Dendrite v2 is an OpenClaw context engine plugin that segments conversations by topic, scores segments for relevance each turn, and assembles a context window that always contains the most relevant conversation at the right level of detail. The agent never runs out of context and never loses track of where a topic left off.

## Problem

OpenClaw sessions are long-lived. As conversation grows, context fills linearly until compaction summarizes older messages — losing detail the agent may need later. When the user drops a topic and returns to it hours or days later (after compaction or a session reset), they must manually prompt the agent to rebuild context. There is no mechanism for the agent to automatically recall where a topic left off.

The core insight: the context window is not a container for the entire conversation. It is a viewport onto the parts that matter right now.

## Approach: Progressive Token-Budget Allocation

Dendrite replaces OpenClaw's default context management with a system that:

1. **Segments** conversation into topic-based chunks using LLM drift detection.
2. **Scores** each segment for relevance to the current conversation (semantic similarity + recency).
3. **Allocates** token budgets per segment proportional to relevance — high-relevance segments get full messages, medium get partial expansion, low get just a summary, irrelevant get nothing.
4. **Assembles** a reshaped message array every turn that fits the context window by being selective, not by compacting.

This is progressive: early in a session everything stays fully expanded. As the conversation grows and topics accumulate, less-relevant segments gradually compress. The transcript on disk is never modified. `compact()` never fires. `assemble()` is always selective.

## Architecture

### The Segment

The fundamental unit. Represents a contiguous stretch of conversation about one topic.

```typescript
interface Segment {
  id: string;
  topic: string;                 // "REST API auth", "Docker networking"
  embedding: number[];           // vector for relevance scoring
  messageIds: string[];          // transcript entry IDs (stable, not positional indices)
  messageCount: number;
  tokenCount: number;            // estimated via character-based heuristic (chars / 4)
  summary: string | null;        // generated lazily on first compression
  summaryTokens: number;
  lastActiveAt: number;          // timestamp
  status: "active" | "closed";
}
```

- **Boundary detection**: LLM-based drift detection (reused from v1, validated against real Atlas sessions — 14/14 forks correctly detected). When a new message drifts from the active segment's topic, the segment closes and a new one opens. The v1 `LLMDriftDetector` takes a `messages` array — an adapter extracts messages from the transcript by ID and presents them in the expected format.
- **Maximum segment size**: if the active segment exceeds `maxSegmentMessages` (default 80), it is force-split even without detected drift. This prevents unbounded growth during long single-topic sessions. The split creates a new segment with the same topic label, and the old segment's summary is generated eagerly (since it will be needed soon).
- **Gradual drift**: conversations that evolve slowly over many messages without a clear break point are handled by the max segment size guard. This is an accepted approximation — a 80-message segment may contain gentle topic evolution, but that's preferable to over-segmenting.
- **Summaries are lazy**: generated by a cheap model the first time a segment needs compression. Captures: topic, key decisions, open questions, code/technical artifacts, and how the segment ended (resolved, interrupted, still open).
- **Embeddings**: computed when a segment closes, using OpenClaw's configured embedding model (Gemini) via `api.runtime` if available, or direct Gemini API call with workspace credentials as fallback. The active segment's embedding is recomputed every ~5 messages.

### The Assembly Algorithm

Called every turn via `assemble({ messages })`. Receives the full message array, returns a reshaped array that fits the context window.

**Step 1 — Score segments for relevance**

Each segment gets a relevance score:

```
score = (alpha * semantic_similarity) + ((1 - alpha) * recency_score)
```

- `semantic_similarity`: cosine distance between segment embedding and embedding of the last `queryWindowSize` messages (default 5, configurable).
- `recency_score`: exponential decay based on turns since the segment was last active.
- `alpha`: tunable, default 0.7 (mostly semantic, recency as tiebreaker).
- Uses OpenClaw's existing Gemini embedding model — no new infrastructure.

**Step 2 — Allocate token budgets**

Sort segments by relevance, highest first. Walk down the list allocating from a total budget of `contextWindow - reserveTokens`. The context window size is read from the model configuration via `api.config` at session start; if unavailable, falls back to a conservative default (32k).

When the active segment itself exceeds the available budget (e.g., a 200-message single-topic coding session), its oldest messages are truncated from context. The active segment always gets at least `reserveTokens` worth of its most recent messages — it is never fully compressed.

| Tier | What goes in context | When |
|------|---------------------|------|
| Active segment | Full messages, always | Always — this is the current conversation |
| High relevance | Full messages | While budget allows |
| Medium relevance | Summary + most recent messages that fit remaining budget | Partial budget remaining |
| Low relevance | Summary only | Minimal budget remaining |
| Below cutoff | Nothing | Budget exhausted |

**Step 3 — Build the message array**

1. Summaries of compressed segments are collected into a single `system`-role message at the start of the returned array: `[Prior context — <topic>: <summary>]` blocks, one per segment. This is clearly separated from conversation messages so the model treats it as background context, not active dialogue.
2. Partially-expanded segments get a brief `system`-role context note followed by their recent messages as normal user/assistant turns.
3. The active segment's messages are included in full, unmodified.
4. Return `{ messages, estimatedTokens }`.

### Ingest

Called when new messages enter the session. Runs before `assemble()` — ordering is guaranteed by the context engine contract (ingest then assemble, sequential, no race conditions).

1. If no segments exist (cold start), create the first segment. No drift detection.
2. If the active segment has fewer than `minMessagesBeforeDrift` messages, add unconditionally. Too early for reliable drift detection.
3. If the active segment exceeds `maxSegmentMessages`, force-split (close current, open new with same topic).
4. Otherwise, run drift detection. If drift exceeds threshold: close the active segment (compute embedding), open a new segment. The `driftThreshold` config controls the LLM detector's confidence gate — the detector classifies messages as on-topic or tangent, and this threshold sets the minimum confidence required to trigger a split.
5. Add the message to the active segment.
6. Persist the segment index as a `custom` transcript entry (excluded from model context) only when the index structure changes (new segment created, segment closed, force-split). On regular message additions, update the in-memory index only. This avoids writing hundreds of copies of a growing index to the transcript in long sessions. The assembly log (written every turn) provides per-turn observability.

**Latency**: drift detection requires an LLM call on every user message. Using free OpenRouter models, expect 1-3 seconds added latency per ingest. This runs before the main model call, so it adds to perceived response time. If latency is unacceptable, drift detection can be disabled (all messages go to one segment, and dendrite becomes a pass-through).

### Compact

```typescript
async compact() {
  return { ok: true, compacted: false };
}
```

Never fires. The assembly algorithm is always selective — it never tries to fit everything, so the context window never overflows. The transcript grows indefinitely on disk (cheap JSONL). If the user returns to an old topic, its relevance score rises and it gets pulled back into context.

## Reuse from Dendrite v1

| Component | Status |
|-----------|--------|
| LLM drift detection prompt logic | Reused — the LLM prompt and response parsing from v1's `LLMDriftDetector` are extracted into a standalone function: `detectDrift(recentMessages: Array<{role, content}>, newMessage: string, model, threshold)`. This avoids coupling to v1's heavyweight `BranchNode` type system. The v1 class itself is not reused |
| Topic labeling | Reused — drift detector suggests topic names |
| Discord metadata extraction (`extractDiscordContent`) | Reused — the content-extraction logic from v1's session adapter. The file-reading parts of the adapter are dropped (v2 receives messages from the plugin API, not from disk) |
| Branch tree, fork/merge/prune | Dropped — replaced by flat segment list |
| Return detection (`LLMReturnDetector`) | Dropped — relevance scoring handles this implicitly |
| Context composer | Rewritten — builds full message arrays, not injection blocks |
| Hook-based architecture | Dropped — replaced by context engine plugin |
| `.dendrite/` state files | Dropped — state lives in `custom` transcript entries |

## Error Handling

| Failure | Behavior |
|---------|----------|
| Drift detection fails (timeout, rate limit) | Fail open — treat message as on-topic, log failure |
| Drift model removed/unavailable | Disable drift detection entirely, all messages go to one segment. Dendrite becomes a pass-through. Log warning |
| Drift model returns garbage (bad JSON, refusal) | Same as timeout — fail open, log the raw response for debugging |
| Summary generation fails | Fall back to segment's last few messages as rough proxy, or just topic label: `[Prior context — Docker networking: ~12 messages, summary unavailable]` |
| Embedding model unavailable | Fall back to recency-only scoring (alpha forced to 0) |
| Stale segment index | Rebuild by scanning transcript entries: messages are assigned to segments by matching their IDs against the last known index. Messages with unknown IDs are assigned to a catch-all segment. Full re-segmentation via drift detection is possible but expensive and only used if no index exists at all |
| Active segment exceeds budget | Truncate oldest messages from active segment, keeping at least the most recent `reserveTokens` worth |
| Cold start / very short session | Pass-through — assemble() returns messages unchanged until 2+ segments exist |
| Multiple concurrent sessions | Each session has its own transcript and segment index, no conflicts |

## Observability

- **Segment index as `custom` transcript entries** — queryable, invisible to agent.
- **Per-turn assembly logging** — which segments included, at what detail, relevance scores, budget allocation, fallbacks. Written as `custom` entries.
- **Agent-facing skill** (`dendrite-inspect`) — shows segment list with message counts and token sizes, current relevance scores, context budget breakdown (which segments are expanded/compressed/excluded), and any recent errors.
- **CLI inspector** — reads `custom` entries from transcript externally.
- **Gateway logs** — errors and fallbacks via `api.logger`.

## Scope Boundaries

Explicitly not in scope for v2:

- **Cross-session context** — dendrite operates within a single session. Memory system handles cross-session persistence.
- **Agent-initiated topic switching** — dendrite ensures the right context is visible when the user brings up a topic. It doesn't proactively nudge the agent about unfinished threads.
- **Multi-agent coordination** — subagents run their own sessions with their own segment state.
- **Modifying message content** — assemble() reshapes which messages are visible, never alters content.
- **Replacing the memory system** — dendrite manages the context window. `memory_search`, `MEMORY.md`, and daily logs remain untouched.

## Design Decisions

### Why no explicit return detection?

Dendrite v1 had an `LLMReturnDetector` that explicitly classified "is the user returning to a previous topic?" This is dropped in v2 because relevance scoring handles it implicitly: when the user says "back to the Docker thing," the embedding of that message has high cosine similarity to the Docker networking segment, its relevance score rises, and it gets pulled into context on the next assemble() call.

Risk: very short or vague return references ("let's go back to what we were doing before") may not embed well against any prior segment. This is an accepted limitation for v2. If it proves problematic in practice, an explicit return classifier can be re-added as a secondary signal in the scoring formula.

### Why flat segments instead of a tree?

Conversations don't have a natural hierarchy. The first topic discussed isn't more "main" than the third. A tree structure (v1's approach) implies parent-child relationships that don't exist in practice. Flat segments with relevance scoring let topics compete on equal footing — the most relevant topic gets the most context, regardless of when it was discussed.

### Plugin info field

The context engine factory returns an `info` object:

```typescript
info: {
  id: "dendrite",
  name: "Dendrite",
  ownsCompaction: true  // tells OpenClaw that dendrite handles compaction
}
```

`ownsCompaction: true` prevents OpenClaw from running its own compaction logic. Since `compact()` always returns `{ ok: true, compacted: false }`, this effectively disables compaction — which is the desired behavior, as assemble() is always selective.
